1.1

What Is Machine Learning?

The Algorithm Paradigm — and Its Limits

At the core of traditional computer programming lies the algorithm — a precise, finite sequence of instructions that transforms a given input into a desired output. Since computers were first built, programmers have been able to devise algorithms for an enormous range of tasks, making computers indispensable in professional and social life.

⚠ The Fundamental Problem
For some tasks, despite decades of research, no algorithm exists. These are often tasks humans perform effortlessly and unconsciously — such as recognising a face, navigating a crowded room, playing chess, driving a car, or holding a conversation in a foreign language. Because we do these things without being aware of how, we cannot write down the procedure.

The Machine Learning Approach

Machine learning's key insight is: instead of writing the algorithm explicitly, we learn it from data. The approach is:

  1. Start with a very general model that has many free parameters. Depending on parameter settings, this model can perform all sorts of tasks.
  2. Learning = adjusting parameter values so that the model best matches the observed training data.
  3. After training, the model with its particular parameter setting becomes specialised to the task underlying the data — it effectively becomes the algorithm for that task.
✦ Core Definition
Machine learning is programming computers to optimise a performance criterion using example data or past experience. A model is defined up to some parameters, and learning is the execution of a computer program to optimise those parameters using training data.

Why Now? The Data Explosion

Today we have many problems for which we lack algorithms but possess vast amounts of data. Digital technology has made data generation automatic and ubiquitous — every product purchase, movie rental, web visit, social media post, and physical commute generates data. We are simultaneously generators and consumers of data: we want products and services personalised to us, which requires learning our needs and predicting our interests.

The Supermarket Example

Consider a supermarket chain selling thousands of goods to millions of customers, recording each transaction (date, customer ID, items, quantities, total spend). The chain wants to learn rules of customer behaviour — predicting which customer will likely buy which product — to maximise sales and profit. Similarly, each customer wants to find products matching their needs.

Customer behaviour is not completely random (people buy beer with chips, ice cream in summer, hot chocolate in winter), but the complete process is too complex to specify. We believe we can construct a good and useful approximation that captures certain patterns or regularities, which can then be used for future predictions — assuming the near future resembles the past.

Data Mining

The application of machine learning methods to large databases is called data mining. The analogy: just as raw earth is mined to extract precious material, a large volume of data is processed to construct a simple model with valuable predictive accuracy.

Application domains: Retail, Finance (credit scoring, fraud detection, stock market), Manufacturing (optimisation, control, troubleshooting), Medicine (diagnosis), Telecommunications (network optimisation), Science (physics, astronomy, biology data analysis), and the World Wide Web (search, relevance).

ML as Part of Artificial Intelligence

Machine learning is not just a database problem — it is also a part of AI. For a system to be genuinely intelligent, it must be able to learn and adapt to environmental changes. Without learning, the system designer must foresee and hard-code solutions for every possible situation — an impossible task in complex, changing environments.

Pattern Recognition Example — Face Recognition

Humans recognise faces effortlessly but cannot explain how. A face is not a random collection of pixels; it has structure (symmetry, eyes, nose, mouth in specific positions). Each person's face is a distinct pattern of these features. A learning program analyses sample face images, captures the pattern specific to that person, and then recognises by checking for this pattern in a new image. This is an instance of pattern recognition.

The Role of Statistics and Computer Science

ML uses statistics to build mathematical models because the core task is making inferences from a sample. Computer science plays two roles:

  • Training: efficient algorithms to solve the optimisation problem, and to store/process massive data.
  • Inference: once a model is learned, its representation and algorithmic solution for inference must be efficient.

In some applications, the space and time complexity of the learning/inference algorithm may be as important as predictive accuracy.

Predictive vs Descriptive Models

A model may be:

  • Predictive — to make predictions about unseen future data.
  • Descriptive — to gain knowledge and understanding from data.
  • Both simultaneously.
1.2

Examples of Machine Learning Applications

1.2.1 — Association Rules

In retail (e.g., a supermarket chain), basket analysis finds associations between products purchased by customers:

Rule Form
"If people who buy X typically also buy Y, then a customer who buys X but not Y is a potential Y customer."

Formally, we are interested in learning a conditional probability:

P(Y | X) — probability of buying Y given X was purchased

Example: P(chips | beer) = 0.7
→ 70% of beer buyers also buy chips.

We can refine this further by conditioning on customer demographics D (gender, age, marital status):

P(Y | X, D) — personalised conditional probability

This framework extends beyond supermarkets — booksellers recommend books, streaming services recommend movies, and web platforms recommend links, all using the same conditional probability machinery.

1.2.2 — Classification

Credit scoring is a canonical classification example. A bank must assess the risk that a loan applicant will default. Given customer attributes (income, savings, collaterals, profession, age, past financial history), the bank classifies applicants as low-risk or high-risk.

The bank trains on historical data (past loans + whether they were repaid) to infer a general rule. A learned discriminant might look like:

IF income > θ₁ AND savings > θ₂ THEN low-risk ELSE high-risk

This is a discriminant function — it separates examples of different classes. Geometrically (Figure 1.1 in text), it draws a boundary in the income–savings space.

Probabilistic Output
Instead of a hard 0/1 decision, we may want P(Y | X). E.g., if P(high-risk | x) = 0.8, the customer has 80% probability of being high-risk. The bank then decides based on possible gain vs. loss.

Other Classification Applications

  • Optical Character Recognition (OCR): Input = image of character; classes = characters. Especially hard for handwriting (zip codes, cheques) because the same character has many visual forms.
  • Face Recognition: Input = face image; classes = identities. Harder than OCR — more classes, larger inputs, 3D object with pose/lighting variation, possible occlusion (glasses, beard).
  • Medical Diagnosis: Inputs = patient info (age, gender, history, symptoms); classes = illnesses. Missing inputs are common (tests not yet done). Wrong decisions can have serious consequences, so the classifier may reject and defer to a human expert.
  • Speech Recognition: Input = acoustic signal; classes = words. Temporal input (sequence of phonemes). Acoustic information alone is insufficient — a language model encoding syntactic and semantic rules is critical.

Natural Language Processing Applications

  • Spam Filtering — ongoing adversarial game between spammers and filters.
  • Document Summarisation
  • Trend Extraction from blogs and social media
  • Sentiment Prediction — is a customer happy or unhappy with a product?
  • Machine Translation — providing large example pairs of texts in two languages and learning the mapping automatically. Current translation systems are built this way.

Biometrics

Biometrics = recognition/authentication using physiological (face, fingerprint, iris, palm) and/or behavioural (signature dynamics, voice, gait, keystroke) characteristics. Multiple uncorrelated inputs make forgery harder and the system more accurate. ML is used both in individual recognisers and in combining decisions from multiple modalities.

Knowledge Extraction and Outlier Detection

  • Knowledge extraction: A learned rule is a simple model explaining the data. E.g., knowing the discriminant separating low-risk from high-risk customers gives knowledge about the properties of low-risk customers, enabling targeted advertising.
  • Compression: Fitting a rule to data gives an explanation simpler than the raw data.
  • Outlier / Novelty Detection: Finding instances that do not obey the general rule. Typical instances share characteristics; those that don't are atypical — may indicate fraud or a novel valid case.

1.2.3 — Regression

In regression, the output Y is a real-valued number (not a class label). Example: predicting used car price from attributes (brand, year, engine capacity, mileage, etc.).

Model: y = g(x | θ) — general form

Linear: y = wx + w₀ — w and w₀ are parameters optimised for best fit

Quadratic: y = w₂x² + w₁x + w₀ — when linear is too restrictive
Regression — Linear vs Quadratic Fit

The ML program optimises parameters θ to minimise approximation error — making estimates as close as possible to correct values in the training set. Any nonlinear function can be used.

Other Regression Examples:

  • Robot/Autonomous Car Navigation: Output = steering angle at each time step. Inputs from sensors (camera, GPS). Training data collected by monitoring a human driver.
  • Ranking: Learning a ranking function rather than an absolute value — e.g., ordering movies by predicted user enjoyment based on genre, actors, and past viewing history.
Classification vs Regression
Both are supervised learning problems (input X, output Y, output values provided by supervisor). The difference: Y is a number in regression, and a class code (e.g., 0/1) in classification. The function g(·) is called the regression function in regression and the discriminant function in classification.

1.2.4 — Unsupervised Learning

In unsupervised learning, there is no supervisor providing output labels — only input data. The goal is to find regularities in the input space. Certain patterns occur more often than others; we want to discover what generally happens and what doesn't. In statistics, this is called density estimation.

Clustering

One method for density estimation is clustering — finding natural groupings of input. In customer segmentation, customers similar in their attributes are allocated to the same group, providing the company with natural groupings. Once groups are found, the company can devise strategies specific to different groups (customer relationship management). Outliers in clustering may indicate a niche market.

Document Clustering

Group similar documents (news articles into politics, sports, fashion, arts, etc.). Documents are typically represented as bag of words: a predefined lexicon of N words, each document as an N-dimensional binary vector (1 if word i appears, 0 otherwise). Suffixes removed to avoid duplicates; stop words ("of", "and") excluded. Documents are grouped based on shared words.

1.2.5 — Reinforcement Learning

In some applications, the output is a sequence of actions. No single action matters independently — what matters is the policy (the sequence of correct actions to reach the goal). An action is good only if it is part of a good policy.

✦ Key Concept
Reinforcement learning algorithms learn from past good action sequences to generate good policies. The algorithm assesses the goodness of policies and improves over time.

Game Playing: Chess has few rules but is very complex (large branching factor, long games). A single move is not inherently good or bad — it depends on context. Learning to play games well is hoped to generalise to applications with evident economic utility.

Robot Navigation: A robot searches for a goal location by trying sequences of moves. After multiple trial runs, it learns the correct action sequence from initial to goal state, as quickly as possible while avoiding obstacles.

Challenges in RL:

  • Partial observability: Unreliable and partial sensory information (e.g., a camera-equipped robot doesn't know its exact room location, only that there is a wall to its left).
  • Multi-agent settings: Multiple agents must interact and cooperate (e.g., a team of robots playing soccer).
20.1

Introduction to ML Experiments

Given multiple applicable learning algorithms for a problem, we face two key questions:

  1. How do we assess the expected error of a learning algorithm on a problem? (e.g., can we say with confidence that the error rate will be less than 2%?)
  2. Given two learning algorithms, how do we say which one has less error for a given application? (e.g., MLP with 4 vs 8 hidden units; what is the best k in k-NN?)
⚠ Cannot Use Training Error
Training set error is always smaller than test set error — by definition, the model has seen training data. Also, a more complex model (more parameters) will almost always produce fewer training errors than a simpler one. Therefore, training errors cannot be used to assess or compare algorithms.

Need for a Validation Set

We need a validation set separate from the training set. Even a single run over a validation set may not be sufficient, for two reasons:

  • Small/noisy sets: Training and validation sets may contain exceptional instances (noise, outliers) that mislead us.
  • Algorithm randomness: Methods like backpropagation converge to the nearest local minimum, so starting from different initial weights gives different final classifiers with different validation error rates.

We therefore want several runs to average over sources of randomness. Using the same algorithm multiple times, we generate multiple learners, test them on multiple validation sets, and record a sample of validation errors. We base evaluation on the distribution of these validation errors.

Important Conditions

  1. Conclusions are conditioned on the dataset — we are comparing algorithms on a specific application, not in a domain-independent way. As stated by the No Free Lunch Theorem (Wolpert 1995): there is no universally best learning algorithm. For any algorithm, there is a dataset where it excels and another where it fails.
  2. The division into training/validation pairs is only for testing. Once decisions are made, train the final learner on all labeled data.
  3. Validation sets used for testing (e.g., algorithm selection) effectively become part of the data used. A separate test set — never seen during training or validation — must be kept for final error reporting. Typically: leave 1/3 as test set, use 2/3 for cross-validation.
  4. Comparison criteria beyond error rate: risk (loss functions), training time/space complexity, testing time/space complexity, interpretability (knowledge extraction), and programmability.
Role of each set:
• Training set → optimise model parameters
• Validation set → optimise hyperparameters / model structure
• Test set → report final error (used only once, at the very end)

Example (MLP):
• Training → optimise weights
• Validation → decide # hidden units, learning rate, stopping point
• Test → report final error of chosen architecture
20.2

Factors, Response, and Strategy of Experimentation

As in other branches of science and engineering, in ML we do experiments to get information about the process under scrutiny — a learner that, having been trained on a dataset, generates an output for a given input.

Key Terminology

  • Experiment: A test or series of tests where we manipulate factors and observe changes in the response.
  • Factors: Variables we vary — can be the algorithm, training set, input features, hyperparameters, etc.
  • Response: The measured output — e.g., classification accuracy on a test set.

Goals of experimentation: identify the most important factors, screen unimportant ones, or find the configuration that optimises the response.

Controllable vs Uncontrollable Factors

TypeDefinitionExamples
ControllableFactors we have control overLearning algorithm, hyperparameters (hidden units, k, C), dataset used, input representation
UncontrollableFactors adding undesired variabilityData noise, particular training subset (in resampling), randomness in optimisation (initial weights in gradient descent)

Strategies of Experimentation (Figure 20.2)

Strategy A: Best Guess

Start at a configuration believed to be good. Test response, fiddle with one (or very few) factors at a time until reaching a "good enough" state. Drawback: no systematic approach; no guarantee of finding the best configuration.

Strategy B: One Factor at a Time (OFAT)

Set a baseline for all factors, then try different levels for one factor while keeping all others at baseline. Major disadvantage: assumes no interaction between factors, which is usually false. For example, in a PCA + k-NN cascade, each choice for dimensionality d defines a different input space where a different k may be optimal.

Strategy C: Factorial Design (Grid Search)

The correct approach: vary all factors together. All combinations of factor levels are tested.

OFAT time complexity: O(L · F) — L levels, F factors
Factorial time complexity: O(L^F) — exponentially more runs, but captures interactions

This "colloquially called grid search" is necessary when factors interact — which is most of the time.

20.3

Response Surface Design

A full factorial design with many factors and levels is expensive. Two alternatives exist:

  • Fractional factorial design: Run only a subset of all combinations.
  • Response surface design: Use knowledge from previous runs to estimate which configurations are likely to have high response.

The Quadratic Assumption

If we can assume the response is quadratic (with a single maximum), we don't need to try all levels. Instead:

  1. Start with some initial runs.
  2. Fit a quadratic function to the results.
  3. Find the maximum of the fitted quadratic analytically.
  4. Take that as the next estimate, run an experiment there.
  5. Add new data to the sample, refit, repeat — until convergence.

General Response Surface Model

r = g(f₁, f₂, …, f_F | φ)

where:
r = response variable
fᵢ = factor i (i = 1, …, F)
φ = parameters of the fitted model
g(·) = typically a quadratic regression function

Uncontrollable factors are modelled as noise.

After a small number of runs around some baseline (defined by a design matrix), enough data exists to fit g(·). We analytically find where g is maximised, run an experiment there, add the data, refit, and iterate until convergence.

⚠ Limitation
This approach works only if the response can indeed be written as a quadratic function of the factors with a single maximum. If the response surface is multimodal or highly non-convex, this method may converge to a local rather than global optimum.
20.4

Randomization, Replication, and Blocking

These are the three basic principles of experimental design.

1. Randomization

The order in which experimental runs are carried out should be randomly determined so that results are independent. In physical experiments, machines need warm-up time, so random ordering prevents time from biasing results.

ML Note
Ordering is generally not a problem in software/ML experiments. However, randomisation is still important in which subsets of data are used for training vs validation.

2. Replication

For the same configuration of controllable factors, the experiment should be run multiple times to average over the effect of uncontrollable factors. In ML, this is done by running the same algorithm on multiple resampled versions of the same dataset — which is cross-validation.

How the response varies across replications gives an estimate of experimental error (effect of uncontrollable factors), which is needed to determine how large a difference must be to be statistically significant.

3. Blocking

Blocking reduces or eliminates variability due to nuisance factors — factors that influence the response but are not of interest.

ML Context
When comparing two learning algorithms using resampling, both algorithms must use the same resampled subsets. Otherwise, accuracy differences could be due to different training data, not the algorithms themselves. Making them use identical training sets is blocking. In statistics for two populations, this is called pairing and is used in paired testing.
20.5

Guidelines for Machine Learning Experiments

These guidelines apply regardless of task type (classification, regression, unsupervised, reinforcement learning). The difference is only in the sampling distribution of the collected response data.

A. Aim of the Study

State the problem clearly and define objectives:

  • Assess expected error of a learning algorithm (e.g., is error < acceptable level?).
  • Determine which of two algorithms has less generalisation error.
  • Choose the best among more than two algorithms, or order them by error.
  • Compare algorithms across multiple datasets.

B. Selection of the Response Variable

Decide the quality measure:

  • Misclassification error for classification.
  • Mean square error (MSE) for regression.
  • Risk — generalisation of 0/1 loss using an arbitrary loss function.
  • Precision and Recall for information retrieval.
  • In cost-sensitive settings: system complexity, not just output.

C. Choice of Factors and Levels

Factors depend on the aim: hyperparameters if optimising a fixed algorithm; the algorithm itself if comparing algorithms; datasets if doing multi-dataset comparison.

Key advice on levels: Normalise factor levels. For k-NN, try k = 1, 3, 5, … For Parzen window bandwidth h, don't use absolute values — instead, express h as multiples of a data-derived statistic (e.g., average distance to nearest neighbour), since absolute values depend on input scale.

D. Choice of Experimental Design

  • Use factorial design unless sure that factors don't interact (they usually do).
  • Replication number depends on dataset size; keep small for large datasets. Too few replicates make distribution comparison difficult; Gaussianity assumptions may not hold for parametric tests.
  • Leave some data as the test set before any experimentation; use the rest for training/validation via resampling.
  • Avoid toy/synthetic data. Use datasets from real-world conditions. Low-dimensional didactic examples can provide intuition but algorithm behaviour in high-dimensional spaces may differ completely.

E. Performing the Experiment

  • Do trial runs at a few random settings before the full factorial experiment to check all is as expected.
  • Save intermediate results (or RNG seeds) so parts can be rerun. All results should be reproducible.
  • Be aware of software aging effects in long experiments.
  • Be unbiased: investigate your own algorithm and competitors equally diligently. In large-scale studies, consider having different people for testing and development.
  • Use reliable, well-tested code libraries rather than writing custom implementations. Follow good software engineering practices: documentation, version control.

F. Statistical Analysis of the Data

Frame questions as hypothesis tests. E.g., "Is A more accurate than B?" becomes "Can we say the average error of learners trained by A is significantly lower than those trained by B?"

Use visual analysis: histograms of error distributions, whisker-and-box plots, range plots.

G. Conclusions and Recommendations

  • Most statistical studies are iterative. Invest no more than 25% of resources in the first experiment — first runs are for investigation only.
  • Statistical testing tells us how much the sample concurs with the hypothesis — not that the hypothesis is definitively true or false.
  • When expectations are not met, investigate why. A deficiency in your favourite algorithm is a hint for improvement.
  • Do not move to the next step before completely analysing the current data. Test every idea before acting on it.
20.6

Cross-Validation and Resampling Methods

For replication, we need multiple training/validation pairs from dataset X (after setting aside a test set). Ideally, divide X into K parts, divide each into training and validation halves — but datasets are never large enough for this. So we use cross-validation: repeated use of the same data, split differently each time.

⚠ Dependency Issue
Cross-validation makes error percentages dependent because different sets share data. This is a known limitation we must account for in statistical testing.

Goals when generating K training/validation pairs:

  • Keep training and validation sets as large as possible (robust error estimates).
  • Keep overlap between different sets as small as possible.
  • Stratification: Ensure classes are represented in their correct proportions in each sample. If a class has 20% examples overall, each sample should also have ~20%.

20.6.1 — K-Fold Cross-Validation

Dataset X is divided randomly into K equal parts X₁, X₂, …, Xₖ. Each pair is formed by holding out one part as the validation set and combining the remaining K−1 as the training set:

V₁ = X₁, T₁ = X₂ ∪ X₃ ∪ … ∪ Xₖ
V₂ = X₂, T₂ = X₁ ∪ X₃ ∪ … ∪ Xₖ
⋮
Vₖ = Xₖ, Tₖ = X₁ ∪ X₂ ∪ … ∪ Xₖ₋₁
K-Fold Cross-Validation — K = 5 (click a row to highlight)

Each fold rotates the validation segment (gold). Training = K−1 parts (blue). Any two training sets share K−2 parts.

Problems with K-Fold

  • To keep training sets large, validation sets become small.
  • Any two training sets share K−2 parts (significant overlap).

Choosing K

  • Typical values: K = 10 or K = 30.
  • As K increases: larger training sets → more robust estimators, but smaller validation sets and higher computational cost.
  • As N (dataset size) increases, K can be smaller. If N is small, K should be large.

Leave-One-Out (LOO)

Extreme case of K-fold: each validation set contains exactly one instance, training uses the remaining N−1. This gives N pairs. Typically used in medical diagnosis where labeled data is scarce. Does not permit stratification.

Modern Practice
With cheaper computation, multiple runs of K-fold cross-validation (e.g., 10 × 10-fold) are now feasible, and averaging over averages gives more reliable error estimates (Bouckaert 2003).

20.6.2 — 5 × 2 Cross-Validation (Dietterich 1998)

Uses training and validation sets of equal size. Procedure:

  1. Divide X randomly into two halves X₁⁽¹⁾ and X₁⁽²⁾. First pair: T₁ = X₁⁽¹⁾, V₁ = X₁⁽²⁾. Second pair (swap): T₂ = X₁⁽²⁾, V₂ = X₁⁽¹⁾. This is fold 1.
  2. Shuffle X randomly, divide into X₂⁽¹⁾ and X₂⁽²⁾, swap to get fold 2's two pairs.
  3. Repeat for 5 folds total → 10 training/validation pairs.
T₁ = X₁⁽¹⁾, V₁ = X₁⁽²⁾
T₂ = X₁⁽²⁾, V₂ = X₁⁽¹⁾
T₃ = X₂⁽¹⁾, V₃ = X₂⁽²⁾
T₄ = X₂⁽²⁾, V₄ = X₂⁽¹⁾
⋮
T₉ = X₅⁽¹⁾, V₉ = X₅⁽²⁾
T₁₀= X₅⁽²⁾, V₁₀= X₅⁽¹⁾

Why 5 folds (not more)? After 5 folds, sets share too many instances — validation error rates become too dependent and add no new information. With fewer than 5 folds, not enough data to fit a distribution and test hypotheses. 5 folds (10 pairs) is the sweet spot.

20.6.3 — Bootstrapping

An alternative to cross-validation: bootstrap generates new samples by drawing N instances from the original dataset of size N with replacement. The original dataset is used as the validation set.

P(instance NOT picked in one draw) = 1 − 1/N
P(instance NOT picked in N draws) = (1 − 1/N)^N ≈ e⁻¹ ≈ 0.368

⟹ Training data contains ~63.2% of instances
⟹ ~36.8% of data is never seen in training
⟹ Error estimate is pessimistic
Bootstrap — Instance Selection Probabilities

Solution to pessimism: replication — repeat many times and average. Bootstrap samples overlap more than cross-validation samples, making estimates more dependent; but bootstrapping is considered the best resampling method for very small datasets.

✦ Authentication Example
Voice login system: false positive = impostor wrongly logged in (much worse). False negative = valid user refused. The two error types are not equally bad. This is why we need separate tp-rate and fp-rate measures.

The ROC Curve

A classifier returning probability P̂(C₁|x) classifies as positive if P̂(C₁|x) > θ. Different values of θ trade off tp-rate against fp-rate:

  • θ close to 1: rarely classify as positive → few false positives, but also few true positives.
  • Decrease θ to increase true positives, but risk more false positives.

Plotting (fp-rate, tp-rate) pairs for different θ values gives the Receiver Operating Characteristics (ROC) curve.

ROC Key Facts
  • Ideal classifier: tp-rate = 1, fp-rate = 0 → upper-left corner of ROC space.
  • Diagonal: making as many true decisions as false ones — worst possible classifier. Any classifier below the diagonal can be improved by flipping its decision.
  • Different θ values correspond to different loss matrices.
  • If one curve is entirely above another, that classifier is strictly better.
  • If curves intersect, classifiers are better under different loss conditions.
  • AUC (Area Under Curve): single summary of ROC performance. Ideal AUC = 1. Compare AUC values across classifiers for general performance averaged over loss conditions.

Precision–Recall in Information Retrieval

A database query retrieves a subset of records. Relevant records exist; retrieved records may or may not be relevant:

  • Precision: retrieved ∩ relevant / total retrieved. (Precision = 1 → all retrieved records are relevant, but some relevant records may be missed.)
  • Recall: retrieved ∩ relevant / total relevant. (Recall = 1 → all relevant records are retrieved, but some irrelevant ones may also be retrieved.)

For different threshold values, one can draw a precision vs. recall curve analogous to the ROC curve.

Sensitivity and Specificity

  • Sensitivity = tp-rate = recall = tp/p.
  • Specificity = tn/n = 1 − fp-rate. How well we detect negatives.
  • A sensitivity vs. specificity curve can also be drawn using different thresholds.

Multi-Class Confusion Matrix

For K > 2 classes with 0/1 error: a K×K matrix where entry (i, j) = number of instances from class Cᵢ assigned to Cⱼ. Ideally all off-diagonal entries are 0. The confusion matrix reveals which pairs of classes are most often confused. Alternatively, define K separate two-class problems, each separating one class from the other K−1.

20.8

Interval Estimation

A point estimator (e.g., maximum likelihood estimator) specifies a single value for a parameter θ. An interval estimator specifies an interval within which θ lies with a certain degree of confidence.

Known Variance Case — Z-Distribution

Given sample X = {xᵗ}ᵗ₌₁ᴺ from N(μ, σ²) with known σ², the sample mean m = Σxᵗ/N is normally distributed: m ~ N(μ, σ²/N).

Define the standard normal statistic:

Z = √N · (m − μ) / σ ~ N(0, 1) …(20.1)

Since 95% of the standard normal lies in (−1.96, 1.96):

P{ −1.96 < √N·(m−μ)/σ < 1.96 } = 0.95

⟹ P{ m − 1.96σ/√N < μ < m + 1.96σ/√N } = 0.95

95% two-sided confidence interval for μ: (m − 1.96σ/√N, m + 1.96σ/√N)

General Two-Sided Confidence Interval

Define zα such that P{Z > zα} = α. By symmetry of Z around 0: z₁₋α/₂ = −zα/₂. General formula:

P{ −zα/₂ < Z < zα/₂ } = 1 − α …(general)

⟹ 100(1−α)% two-sided CI: ( m − zα/₂·σ/√N, m + zα/₂·σ/√N ) …(20.2)

Examples:
95% CI: α = 0.05, zα/₂ = 1.96
99% CI: α = 0.01, zα/₂ = 2.58 ← wider interval for more confidence
Interval shrinks as N increases.

One-Sided Confidence Interval

Since P{Z < 1.64} = 0.95:
P{ m − 1.64σ/√N < μ } = 0.95

(m − 1.64σ/√N, ∞) is the 95% one-sided upper CI for μ (defines lower bound)

General 100(1−α)% one-sided CI: P{ m − zα·σ/√N < μ } = 1−α …(20.3)

Unknown Variance Case — t-Distribution

When σ² is unknown, plug in the sample variance:

S² = Σₜ (xᵗ − m)² / (N − 1)

When xᵗ ~ N(μ, σ²), (N−1)S²/σ² ~ χ²(N−1), m and S² are independent.

Therefore: √N(m − μ)/S ~ t_{N−1} …(20.4)

100(1−α)% two-sided CI:
P{ m − tα/₂,N₋₁ · S/√N < μ < m + tα/₂,N₋₁ · S/√N } = 1−α
t vs Z
The t-distribution has longer tails (larger spread) than the standard normal. Therefore the t-based confidence interval is wider — reflecting additional uncertainty from estimating an unknown variance. As N → ∞, t_{N−1} → N(0,1).

20.9

Hypothesis Testing

Instead of estimating parameters, we test specific hypotheses about them. We define a statistic that follows a certain distribution if the hypothesis is correct. If the statistic calculated from our sample has very low probability under this distribution, we reject the hypothesis; otherwise we fail to reject it.

⚠ Important Nuance
We never say a hypothesis is "true" or "false" — only that the sample appears consistent (or inconsistent) with it to a given degree of confidence.

Null and Alternative Hypotheses

H₀ : μ = μ₀ (null hypothesis — what we test against)
H₁ : μ ≠ μ₀ (alternative hypothesis — two-sided)

OR one-sided:
H₀ : μ ≤ μ₀ vs H₁ : μ > μ₀

The sample mean m is the point estimate of μ. We reject H₀ if m is too far from μ₀.

Two-Sided Z-Test (Known Variance)

Fail to reject H₀ at significance level α if:
√N(m − μ₀)/σ ∈ (−zα/₂, zα/₂) …(20.5)

Equivalently: μ₀ lies within the 100(1−α)% confidence interval.

Type I and Type II Errors

Decision / TruthFail to Reject H₀Reject H₀
H₀ is TrueCorrect ✓Type I Error (α)
H₀ is FalseType II Error (β)Correct (Power) ✓
  • Type I error (α): Reject H₀ when it is correct. α is set before the test (typical values: 0.1, 0.05, 0.01) and defines the tolerated false rejection rate.
  • Type II error (β(μ)): Fail to reject H₀ when the true mean μ ≠ μ₀.
  • Power function: 1 − β(μ) = probability of correctly rejecting H₀ when the true mean is μ.
β(μ) = Pμ{ −zα/₂ ≤ (m − μ₀)/(σ/√N) ≤ zα/₂ } …(20.6)

Type II error increases as μ approaches μ₀ (harder to detect small differences).
Use power analysis to determine required sample size N to detect a difference
δ = |μ − μ₀| with sufficient power.

One-Sided Test

H₀: μ ≤ μ₀ vs H₁: μ > μ₀

Fail to reject if:
√N(m − μ₀)/σ ∈ (−∞, zα) …(20.7)

Key rule: put the claim you want to support in H₁ — if the test rejects, it supports your claim.

Two-Sided t-Test (Unknown Variance)

Fail to reject H₀: μ = μ₀ at level α if:
√N(m − μ₀)/S ∈ (−tα/₂,N₋₁, tα/₂,N₋₁) …(20.9)

This is the two-sided t-test. A one-sided t-test is defined analogously by using a one-sided critical region.

Application to ML Experiment Error Rates

Section 20.10 (not fully in excerpts) extends this framework to testing classification error rates using the binomial distribution (since each prediction is a Bernoulli trial — correct or incorrect). The same methodology applies for squared error in regression, log likelihoods in unsupervised learning, and expected reward in RL — provided the appropriate parametric form for the sampling distribution can be written. Nonparametric tests are used when no suitable parametric form exists.
